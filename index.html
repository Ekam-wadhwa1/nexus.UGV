EW-FRT-2026Section 01

Architectural Paradigm

Modern disaster response infrastructure requires isolated, deterministic mechatronics components paired with high-level cognitive computing nodes. This iteration of the Flood Response Tank achieves this goal by splitting workloads across two distinct processors: a LEGO MINDSTORMS EV3 block and a Raspberry Pi 4 single-board computer.

System Architecture Separation

The Reactive Layer (EV3): Manages real-time physical processes. This block parses control signals and generates motor power profiles instantly, preventing delayed reactions or mechanical freezes.

The Cognitive Layer (Raspberry Pi): Manages heavy data streams, processing visual fields via a DJI Action 2 camera and serving telemetry instantly to an external user dashboard.

By segregating real-time motor functions away from data-heavy network operations, the platform guarantees that the chassis maintains structural stability and maneuverability even when local network bandwidth fluctuates.

Locomotion Kinematics

Navigating loose mud, debris fields, and standing water requires a highly tailored differential-drive track layout. Unlike standard wheeled platforms, continuous tracks distribute structural load across a vastly larger surface area, dramatically reducing soil compaction and sinking risks.

The vehicle's physical movement is governed by differential drive mathematics. The chassis's angular velocity vector ($\omega$) and linear velocity vector ($v$) are determined directly by the independent speeds of the left track ($v_L$) and right track ($v_R$), where $W$ represents the chassis track width:

v = (v_R + v_L) / 2
ω = (v_R - v_L) / W
+-----------------------------------+ | [Track L] --> Speed: v_L | | | | | v Chassis Width (W) | | | | | [Track R] --> Speed: v_R | +-----------------------------------+

When the system achieves $v_L = -v_R$, the vehicle executes a perfect zero-radius spin around its geometric center. This movement profile yields unparalleled maneuverability within confined structural collapse zones.

Fluid Mass Mechanics

Deploying heavy robotics in semi-aquatic environments requires aggressive management of buoyancy forces and center-of-mass vectors. Governed by Archimedes' Principle, any object submersed in fluid experiences an upward buoyant force equivalent to the weight of the displaced fluid.

The buoyant force formula mapped for this chassis is defined as:

F_b = ρ * V * g

Where $\rho$ represents fluid density (approximated at $1000\text{ kg/m}^3$ for water), $V$ represents the submerged volume of the lower hull, and $g$ represents standard gravity ($9.81\text{ m/s}^2$).

Static Equilibrium Metrics

To prevent catastrophic tipping on slick inclines, the center of mass is artificially lowered. Heavy components—specifically the high-torque drive motors and custom lithium cell arrays—are anchored directly to the base plates of the primary structural frame.

This layout ensures the physical center of mass remains well below the center of buoyancy, resulting in a self-righting torque that stabilizes the tank during sudden depth shifts.

Structural Stress Dynamics

The structural integrity of the main body must routinely withstand extreme torsional forces from the dual high-torque motors, alongside severe shear stress from rough terrain navigation. Torsional stress ($\tau$) on the main drive shafts is calculated using the applied torque ($T$), shaft radius ($r$), and the polar moment of inertia ($J$):

τ = (T * r) / J

To mitigate shear stress concentrations, the primary framework utilizes a dense, interconnected cross-bracing matrix. This engineering choice safely distributes mechanical strain across the entire chassis geometry rather than localizing it at individual load-bearing pins.

EW-FRT-2026Section 05

Power Subsystem Metrics

The electrical architecture isolates power delivery into two heavily shielded, completely independent power grids. This prevents electrical noise and voltage spikes generated by the brushed motors from disrupting the sensitive microprocessor logic arrays.

Dual-Rail Power Distribution

Locomotion Power Supply: A high-discharge 7.4V lithium-polymer cell delivers raw energy exclusively to the EV3 motor controller, driving the tracks and the mechanical turret assembly.

Cognitive Power Supply: A dedicated 5V, 3A DC-to-DC step-down converter draws power from an isolated secondary battery pool, providing incredibly clean, stable voltage directly to the Raspberry Pi 4.

This hard-line physical isolation guarantees that severe voltage sags—which frequently occur during high-load mechanical events like climbing out of deep mud—will never trigger an undervoltage reboot on the main communications server.

Dual-Brain Topology

Data transmission across this platform utilizes a custom-engineered pipeline relying on two separate frequency paths: a low-latency wireless Bluetooth mesh for operator inputs, and a high-bandwidth physical USB backbone for local inter-board routing.

[Operator Controller] --- (Bluetooth Loop) ---> [Reactive EV3 Node] | (Direct USB Link) v [External Dashboard] <--- (Wi-Fi Stream) ------ [Cognitive Pi Node]

The handheld operator remote continuously samples physical analog inputs, compiles them into data frames, and broadcasts them to the onboard EV3 chassis node via Bluetooth. The EV3 consumes these instructions to fire the motors, while simultaneously relaying live operational telemetry up the USB backbone to the Raspberry Pi 4.

Master Handheld Input Array

Constructed within the native EV3 Home Edition environment, the handheld operator unit runs a rapid, continuous execution loop designed to read analog lever positions and securely transmit them over the airwaves.

EV3-G Controller Loop Architecture

1. Left Track Telemetry: A rotation sensor block reads the Port A lever angle. The output is routed through a scale multiplier (Math Block / 2) for precision dampening, then beamed via Bluetooth into the "LeftDrive" mailbox.

2. Right Track Telemetry: An identical, parallel process reads the Port D lever, dampens the analog curve, and beams the output integer directly to the "RightDrive" mailbox.

3. Payload Triggers: Front-facing tactile buttons are polled constantly. D-pad horizontal strikes send "LEFT" or "RIGHT" strings to the "Turret" mailbox, while a center strike drops a binary True logic flag into the "Fire" mailbox.

EW-FRT-2026Section 08

Central Processing Router

The onboard chassis node operates as an ultra-low-latency receiver. It sits in an infinite processing loop explicitly designed with three parallel tracks to execute incoming network commands instantaneously.

EV3-G Receiver Node Logic

Drivetrain Actuation (Path A): Dedicated Bluetooth listener blocks extract numeric values from the "LeftDrive" and "RightDrive" mailboxes, injecting those values straight into the raw PWM power inputs of Large Motors B and C.

Turret Targeting (Path B): A String-based Switch block evaluates the "Turret" mailbox. "LEFT" commands counter-clockwise rotation; "RIGHT" commands clockwise rotation; any other state returns to neutral hold.

Payload Execution (Path C): A Boolean Switch block monitors the "Fire" mailbox. Upon intercepting a True flag, it triggers Large Motor D to rotate exactly 360 degrees, flawlessly cycling the mechanical firing cam one complete iteration.

EW-FRT-2026Section 10

Visual Telemetry Capture

A DJI Action 2 camera module acts as the primary optical sensor, mounted to the turret and tethered via USB-C. The Pi intercepts this raw feed using the OpenCV computer vision library, performing aggressive real-time frame optimization to ensure fluid transmission under poor network conditions.

OpenCV Pipeline Optimization

To eliminate video stuttering over saturated 2.4GHz bands, the OpenCV pipeline forces a down-sample of all optical frames to 640x480 resolution. The matrix is then aggressively encoded to JPEG format with a targeted 70% quality ratio prior to network broadcast.

This specific compression profile drops the individual frame payload to roughly 25 KB. At this weight, the system successfully maintains a 30 FPS stream with under 120ms of glass-to-glass latency, which is critical for remote piloting accuracy.

EW-FRT-2026Section 12

Network Topology Parameters

To guarantee command delivery in electronically noisy environments, the system utilizes a heterogeneous network protocol architecture, separating critical control data from high-bandwidth sensor data.

Network Routing Architecture

Tactical Control Link (Bluetooth RFCOMM): Operates entirely on the 2.4 GHz spectrum using a dedicated serial port profile, guaranteeing the delivery of tiny 20 Hz control packets even in high-interference zones.

Internal Data Bus (USB IPv4 Local Link): Establishes a localized 10.42.0.X subnet exclusively via physical USB cable, securing a zero-packet-loss bridge between the Pi and the EV3 chassis.

External Stream (802.11ac Wi-Fi): Broadcasts the heavy optical and JSON telemetry payloads externally over HTTP port 5000, isolating video lag completely from the motor control arrays.

Launch Checklist & Diagnostics

Rigorous pre-flight procedures are mandatory. Execute this diagnostic sequence sequentially prior to any active field deployment of the physical hardware:

Pre-Deployment Sequence

[ ] Node Initialization: Apply power to the Raspberry Pi gateway. Verify the Status Green LED enters a slow pulse, confirming the OS has booted and is awaiting the data bus handshake.

[ ] Data Bus Sync: Execute the EV3 receiver script on the chassis node. Verify the Pi's Status Green LED switches to solid illumination, confirming the USB bridge is fully established.

[ ] Telemetry Verification: Connect to the web dashboard. Confirm the live optical feed is rendering without artifacting and the JSON telemetry gauges are refreshing at a stable 10 Hz.

[ ] Control Link Pairing: Boot the operator remote. Verify successful Bluetooth pairing with the chassis and conduct a micro-test of turret servos to confirm PWM delivery.

Upon satisfactory completion of all diagnostic gates, the system is classified as armed and ready for full-scale tactical operation.
